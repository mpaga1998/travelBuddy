## Phase 1 — Critical security

Nothing else matters until this is done.

> **Deploy target:** Vercel serverless (`api/`) is the single source of truth.
> The duplicate Express backend (`server/`) was deleted once we unified Phase 1
> across both code paths — see 1.3.

- [x] **1.1** Create `requireAuth` — JWT verification that returns the verified user or 401s. Shipped as `api/lib/requireAuth.ts` (serverless-shaped) and previously `server/middleware/requireAuth.ts` (Express).
- [x] **1.2** Apply `requireAuth` to every protected route (`/api/itinerary`, `/api/itinerary/save`).
- [x] **1.3** Stop trusting client-sent `userId`. Every handler now pulls ownership from `req.user.id` (the verified JWT) and fetches `firstName` server-side from the `profiles` table. In the process, unified on Vercel serverless and deleted `server/` + Express deps.
- [x] **1.4** Switch backend Supabase client to `service_role` key. `api/lib/supabaseServer.ts` now prefers `SUPABASE_SERVICE_ROLE_KEY` (server-only name, no VITE_ prefix), falls back to the legacy `VITE_SUPABASE_SERVICE_KEY` with a warning, and fails closed in production if neither is set. `.env.example` documents the new var. Vercel env var added; RLS fixes applied (profiles SELECT locked to self, pin_bookmarks SELECT narrowed to authenticated).
- [x] **1.5** Frontend sends JWT. `src/features/itinerary/itineraryApi.ts` grabs the session token and attaches `Authorization: Bearer <token>` on both calls. (Shipped alongside 1.3.)
- [x] **1.6** Per-user sliding-window rate limiter on `/api/itinerary` (10/hour). Backed by a `rate_limits` table in Supabase — chosen over Upstash to avoid a new vendor. Shipped as `api/lib/rateLimit.ts` + `supabase/migrations/20260421_add_rate_limits.sql` (applied). Fails **open** on DB error (cost-protection limiter should never lock a paying user out because Postgres hiccuped). Sets `X-RateLimit-*` + `Retry-After` headers.
- [x] **1.7** Request size limits. `api/lib/validateBodySize.ts` rejects bodies >100 KB with 413 before any JSON parse / DB call / OpenAI call. Checks `Content-Length` first, falls back to measuring the parsed body (catches chunked uploads and lying clients). Applied to both `/api/itinerary` and `/api/itinerary/save`.
- [x] **1.8** Verified git history clean across all 173 commits / all 22 refs (local + origin). `.env` never tracked (confirmed via `git log --all -- .env` + `git ls-files`, and it's gitignored at `.gitignore:26`). Pickaxe search (`git log -S`) for the live OpenAI key prefix and the Supabase project ref returned zero hits. `.env.example` contains only `sk-...` / `eyJ...` placeholders — no real values. The worktree `.env` holds real secrets but is properly ignored. If any branch is ever force-pushed or rebased, re-run these checks before merging.

## Phase 4 — Moderation

Before you ever go public. User-generated map + images + no moderation = trouble.

- [x] **4.1** Create `pin_reports` table (`pin_id`, `reporter_id`, `reason`, `created_at`) + cached count column on `pins`.
- [x] **4.2** Add "Report pin" button in the pin popup.
- [x] **4.3** Auto-hide at threshold. `is_hidden` column on pins; trigger or cron sets true when reports ≥ N. Filter from `listPins`.
- [x] **4.4** OpenAI moderation on submission. New `api/lib/moderation.ts` calls `omni-moderation-latest` (free endpoint) with whitespace short-circuit and **fail-open** policy on every error path (no key, network down, parse failure) — same rationale as the rate limiter: a third-party hiccup must never block content creation. Wired into two paths: (1) `api/itinerary.ts` moderates `notes + desiredAttractions` joined into one input, BEFORE Mapbox geocoding / weather / OpenAI generation, returning a 400 with the standard rejection message — flows through the existing form `error` prop, no form changes needed; (2) new `api/moderate.ts` route (POST, requireAuth + validateBodySize, no rate limiter — moderation is free) that the client pre-flights from `MapView.onSubmitDraft` via `src/lib/moderation.ts#checkContentAllowed` before image upload + `createPin`. Locations are intentionally NOT moderated — geographic strings like war-zone place names trip violence categories with high false-positive rate. Both server and client constants for the user-facing message are deliberately generic so we don't enumerate ban-evasion paths.
- [x] **4.5** Admin moderation dashboard at `/admin`. Server: `api/lib/requireAdmin.ts` wraps `requireAuth` and checks `profiles.is_admin = true` via the service-role client (bypasses RLS so the check is authoritative); on failure writes 403 with a generic "Forbidden" message that doesn't confirm/deny admin tooling exists. Two routes: `GET /api/admin/pins` returns `{ reported, hidden }` with author label + 5 most recent report reasons (service-role bypasses RLS on pins/profiles/pin_reports); `POST /api/admin/action` accepts `{ pinId, action: 'hide' | 'restore' | 'delete' }` and logs `[admin <id>] <action> <pinId>` to Vercel for a loose audit trail (delete cascades through FK to pin_reports/reactions/bookmarks). Frontend: `src/features/admin/AdminPage.tsx` with two side-by-side sections (Reported / Hidden), `useConfirm` for destructive prompts, sonner toasts, full re-fetch after every action (queue is small enough, simpler than diffing local state). Routing: pathname-based without adding a router dep — `App.tsx` watches `/admin`, runs `isCurrentUserAdmin()` for signed-in users, renders `AdminPage` on allow and a generic 404 on deny so the URL stays opaque to non-admins; `popstate` listener keeps back/forward working. Admin bootstrap: one-line SQL to flip my own `is_admin = true` (handed to user separately).
- [x] **4.6** Terms of Service + Community Guidelines pages. `/terms` and `/guidelines` are public routes (short-circuit before the auth guard). `src/features/legal/TermsPage.tsx` and `GuidelinesPage.tsx` — amber draft callout, prose sections, governing law (Italy / EU). Signup gate: `agreedToTerms` checkbox in `AuthPage` signup variant; disables "Sign up" until checked; links open pages in a new tab. Footer on `InitialPage` with in-app `popstate`-based navigation (no router dep added).

## Phase 5 — The community layer

This is what makes it a "community" app. Currently missing despite the pitch.

- [x] **5.1** Public user profile pages at `/u/:handle`. Migration `20260428_add_public_profile_fields.sql` adds `profiles.handle` (unique nullable, regex CHECK `^[a-z0-9_-]{3,30}$`, partial unique index for non-null values), `profiles.bio` (nullable, length CHECK ≤ 280), and `itineraries.is_public` (default false). RLS on itineraries widened from owner-only SELECT to `auth.uid() = user_id OR is_public = true` so non-owners only see opt-in itineraries — owners still see their full set. New `src/features/profile/handleValidation.ts` enforces format + a reserved-handle blacklist (admin/api/u/profile/etc.) at the app layer so the list can be tuned without a migration. New `src/features/profile/publicProfileApi.ts` wraps the read paths (`fetchPublicProfileByHandle`, `fetchPublicPinsByUserId`, `fetchPublicItinerariesByUserId`) plus the `setItineraryPublic` toggle and an advisory `isHandleAvailable` (DB unique index is the actual source of truth — UPDATE catches `23505` and surfaces "handle taken"). New `PublicProfilePage.tsx` renders header (avatar, display name, @handle, role pill, bio, pin/itinerary counts), pins grid, and public itineraries list with click-to-expand modal showing markdown. 404 state when the handle doesn't resolve. Routing: `/u/<handle>` parsed in `App.tsx` via regex helper, short-circuited BEFORE the auth guard so the page works signed-out; popstate listener keeps back/forward in sync. Editing: ProfileInfoTab gains a Handle field (with `@` prefix, format hint, "View public profile →" link when matching the saved value) and a Bio textarea (280-char counter); save validates format + uniqueness, surfaces unique-violation 23505 as a friendly error. Sharing: SavedItinerariesTab detail view gains a public/private checkbox (optimistic update with revert-on-failure); list rows get a green "Public" badge. Pin attribution: `Pin.createdByHandle` joined through `pinApi.ts`, PinPopup renders the "Pinned by X" pill as a clickable link to `/u/:handle` when the author has a handle (uses `pushState` + manual `popstate` dispatch to bridge the createRoot-mounted popup back into App's path-based router).
- [x] **5.2** Follow system. Migration `20260428_add_follows.sql` adds the `follows` table (`follower_id`, `followee_id`, `created_at`) with `UNIQUE(follower_id, followee_id)` and a `CHECK (follower_id <> followee_id)` for self-follow prevention, plus indexes in both directions. Cached `followers_count` + `following_count` columns on profiles maintained by SECURITY DEFINER triggers on AFTER INSERT/DELETE — same pattern as the pin_reports trigger from 4.1, so reads stay O(1). RLS: SELECT open (the relationship graph is public, like Twitter/X — required for "is X following Y" checks from the follow button), INSERT/DELETE restricted to `follower_id = auth.uid()`. New `src/features/profile/followApi.ts` with `followUser` (idempotent — swallows 23505 unique-violation), `unfollowUser` (idempotent — RLS makes deleting a non-existent row a no-op), and `isFollowing` (returns false on any failure including unauthenticated, defensive). PublicProfilePage parallelizes the follow check with pin/itinerary fetches; header gains follower/following counts (from cached columns, never aggregated client-side) plus a Follow/Unfollow button hidden on own profile and for signed-out viewers. Optimistic toggle: button + count update locally, revert on API failure with toast. The button shows a "Loading…" state while the initial `isFollowing` check is in flight to avoid flashing the wrong label.
- [x] **5.3** Activity feed at `/feed`. No new schema — leverages the `follows` table from 5.2 and existing `pins` reads. New `src/features/feed/feedApi.ts` runs two queries: `follows.followee_id` for the current user, then `pins WHERE created_by IN (...) AND is_hidden = false` ordered by `created_at DESC`. Two roundtrips beat a PostgREST IN-subquery for clarity and both hit indexed columns. Cursor pagination via `created_at` (`lt(before)` to walk older), page size 20, `nextCursor = null` when the page is short. Distinguishes `noFollows` (empty follow list — different empty state) from "follows but no pins" (followees haven't posted yet). New `src/features/feed/FeedPage.tsx`: header + chronological card stream + Load More button. Each card renders author header (avatar/handle/relative time, clickable to `/u/:handle` via the same pushState + popstate-dispatch trick PinPopup uses), optional hero image, title with category emoji, description clamped to 3 lines, footer with bookmark count + "View on map" button. Inline `formatRelativeTime()` helper prints `just now / Xm ago / Xh ago / Xd ago` and falls back to `Mon DD` past a week — no Intl.RelativeTimeFormat dep needed. Routing: `/feed` added to `App.tsx` BEHIND the auth gate (signed-out visitors hit AuthPage) — back button returns to home, "View on map" pops back and switches into MapView via the existing showInitialPage flag. Discovery: third CTA on InitialPage ("📰 Feed", styled as a translucent secondary pill so it doesn't compete with the headline orange/purple buttons).
- [x] **5.4** Notifications. Migration `20260429_add_notifications.sql` creates the `notifications` table (`recipient_id`, `actor_id`, `kind` ∈ {like, bookmark, follow}, `pin_id` nullable, `created_at`, `read_at`) with a shape CHECK ("pin-related kinds carry a pin_id; follow doesn't") and two indexes — `(recipient_id, created_at desc)` for the inbox query and a partial `(recipient_id) WHERE read_at IS NULL` for the unread badge. Both `recipient_id` and `actor_id` reference `profiles(id)` (not `auth.users`) so PostgREST embedded relations resolve in a single query. Three SECURITY DEFINER triggers — `fn_notify_pin_like` on `pin_reactions` (AFTER INSERT OR UPDATE, fires only when the resulting kind is 'like' AND it's a transition, since toggleReaction in pinApi.ts upserts when switching from dislike → like), `fn_notify_pin_bookmark` on `pin_bookmarks`, `fn_notify_follow` on `follows`. All triggers self-skip same-actor rows and EXIST-check for duplicates so a like → unlike → like sequence yields exactly one notification. RLS: SELECT/UPDATE/DELETE gated to `recipient_id = auth.uid()`; no INSERT policy (only the SECURITY DEFINER triggers write). Frontend: new `src/features/notifications/notificationsApi.ts` (fetchNotifications joining actor profile + pin title, countUnread via head-only count, markAllAsRead optimistic + server-side filtered to `read_at IS NULL`). New `NotificationsPage.tsx` at `/notifications` (signed-in only — same auth-gate placement as `/feed`), renders rows with avatar + sentence + relative time, blue-tinted background + dot for unread, "Mark all read" button when `unreadCount > 0`, click-to-navigate per kind (like/bookmark → map, follow → `/u/:handle`). Bell icon added to InitialPage top-right (left of sign-out) with unread count badge that refreshes on mount + window focus — capped at "99+" so a viral pin doesn't blow up the layout. No Supabase Realtime subscription yet — focus-refresh feels live enough at this scale and avoids the websocket complexity.
- [x] **5.5** Pin comments. Migration `20260430_add_pin_comments.sql` adds `pin_comments` (`id`, `pin_id`, `user_id`, `body`, `created_at`, `updated_at`) with a body length CHECK (1–1000 chars) and indexes on `(pin_id, created_at)` for thread reads + `(user_id)` for "my comments" lookups. `pins.comment_count` cached column maintained by SECURITY DEFINER AFTER INSERT/DELETE triggers (same shape as the other counters). Notifications gain a fourth kind: the migration drops + recreates the `kind` and `pin_id_shape` CHECKs to allow `'comment'`, and `fn_notify_pin_comment` fires on each pin_comments INSERT to notify the pin author (skips self-comments). Comment notifications are intentionally NOT deduped — every comment is a distinct event the author should hear about, unlike like/bookmark which are stateful flags. RLS: SELECT open, INSERT/UPDATE/DELETE limited to `user_id = auth.uid()`. New `src/features/pins/commentsApi.ts` (fetchComments, addComment with client-side body length guard mirroring the DB CHECK, deleteComment). New `PinComments.tsx` mounted inside `PinPopup` after the action row (hidden for ephemeral itinerary pins) with composer (textarea + char counter + Post), inline error surface, optimistic delete with revert, author handle clickable to `/u/:handle` via the existing pushState + popstate trick. Moderation: comment bodies pre-flight through `checkContentAllowed` (the same OpenAI-moderation helper from 4.4) before the insert — fails open on transient errors. NotificationsPage's `describeAction` extended with comment copy ("commented on your pin '..._'"). `Pin.commentCount` joined through `pinApi.ts` SELECT, `seedPins` updated to satisfy the new field.

## Phase 7 — Product wedge

- [ ] **7.1** Pick a niche and commit. Candidates: backpacker/hostel 18-35 (half-built already), solo-travel safety + community, long-term nomad routes. Rewrite landing page and empty states around it.
